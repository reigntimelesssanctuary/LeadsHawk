# v1.9.0 — Two-stage deep scan

**Status:** Spec'd, not yet implemented. Drafted 2026-05-25 by Claude on behalf of
the user, who originated the architecture. Execute as the next release after
v1.8.7.

## Why this exists

Through v1.8.7 the deep scan pipeline became technically robust (no timeouts,
no parse failures, no lazy refusals, no crashes), but the *output quality* still
underwhelms the user. Run #21 logs show both products doing real research
(48–50 citations, 15K completion tokens each) but returning 0 candidates each
— because the model surfaces educational / generic content (industry guides,
standards homepages) instead of specific named-company buying events.

Diagnosis: the current monolithic deep scan asks ONE model (sonar-deep-research)
to simultaneously do all of:

- broad web research
- structured JSON output meeting our schema
- confidence scoring
- per-product signal matching
- ICP fit judgment
- self-skepticism / "most news isn't an opportunity"
- our brand-self hygiene block
- recency filtering
- scan-rule application

Stacked, those constraints push the model toward safe-and-empty. It's a
Ferrari being used to deliver mail.

The user's insight (and the right architecture): split the work into the
phases that mirror how they'd do it manually:

1. **Wide net** — open thematic deep research. Just list named companies in
   relevant situations. No scoring, no filtering, no schema strictness.
2. **Filter the net** — Claude takes that raw list and applies the structured
   judgment: ICP fit, scan rules, dedup, confidence, schema.

Each model does what it's good at.

## Architecture overview

```
runDeepScan()
  └── for each scan-enabled product:
        ├── stage1Discovery(brand, product, recency)
        │     ↓ Perplexity sonar-deep-research, open prompt, loose schema
        │     ↓ Returns: { candidates: [{ company, event, why_relevant,
        │                   source_url, source_title, source_date }] }
        │     ↓ Expected: 10–30 raw candidates
        │
        ├── stage2Qualify(stage1Output, brand, product, ctx)
        │     ↓ Claude Sonnet, structured schema, applies all judgment
        │     ↓ Returns: opportunities[] matching OPPS_SCHEMA
        │     ↓ Expected: 3–10 qualified per product
        │
        └── insertCandidates(qualified, ...)  // existing path, slightly trimmed
```

Manual scan (`sonar-pro`), Live Monitor, brand/product research — all
untouched in v1.9.0. Only `runDeepScan` changes.

## Stage 1 — Discovery (Perplexity)

**Model:** `settings.deepScanModel || 'sonar-deep-research'` via the existing
async API (`completePerplexityAsync`).

**Stage tag:** `'deep_scan_discovery'` (new).

**Token budget:** `maxTokens: 24000` (same as current deep scan).

**Recency:** uses `resolveScanRecency(product, brand)` exactly as the current
deep scan does. Passes through to `search_recency_filter`.

**Prompt structure** (the system + user prompt together, both intentionally
shorter and more open than the current SYSTEM):

```
SYSTEM (~150 words):
You are a senior B2B sales-intelligence research analyst with live web
access. Your job in this stage is RESEARCH AND DISCOVERY only — surfacing
named companies that have recently shown events relevant to a specific
brand+product's target customers.

You do NOT score, filter, or judge. A separate downstream step handles
that. Your job is to cast a wide net.

Mandatory: search the live web with multiple distinct queries. Consult
multiple sources. Cite them.

Be inclusive. Surface 15–30 candidates per call. Better to surface a
weak lead and let the downstream step reject it than to filter it out
here and miss it.

Return JSON in the schema you've been given. Reasoning may be extensive
but the JSON output is mandatory.

USER (per product):
# Brand
[brand.name, category, description, positioning, target_icp,
 competitive_summary, brand-level signals, research_summary truncated to 1200]

# Product
[product.name, category, description, use_cases, differentiators,
 competitors, research_summary truncated to 1500]

# Buying signals to orient your search (NOT a filter — just orientation)
[product.signals bullets]

# Retrieved knowledge
[top 5 knowledge chunks via retrieveRelevantChunks]

# Time window
Only consider events from the last [recency.value].

# Task
Cast a wide net. Identify SPECIFIC NAMED COMPANIES that have recently
shown events relevant to this brand+product's target customers.
Examples of events you should surface:
- Press releases or news about expansion, M&A, lease signings, exec
  changes, capital allocations, regulatory filings
- Job postings indicating organizational moves
- Industry coverage of specific customers in transition

For each candidate, return:
- company: the named organization
- event: what specifically happened (with date if known)
- why_relevant: 1–2 sentences on why this might be a buying signal
- source_url: a real URL we can verify
- source_title: page or article title
- source_date: ISO date or "unknown"

Return 15–30 candidates. Do not filter, score, or judge. The downstream
qualifier will handle that.
```

**JSON schema (loose):**
```json
{
  "type": "object",
  "required": ["candidates"],
  "properties": {
    "candidates": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["company", "event", "source_url"],
        "properties": {
          "company": { "type": "string" },
          "event": { "type": "string" },
          "why_relevant": { "type": "string" },
          "source_url": { "type": "string" },
          "source_title": { "type": "string" },
          "source_date": { "type": ["string", "null"] }
        }
      }
    }
  }
}
```

Note the schema only requires `company`, `event`, `source_url`. Everything
else is optional. Stage 2 will fill in missing fields.

**Return type:**
```ts
export type Stage1Candidate = {
  company: string;
  event: string;
  why_relevant?: string;
  source_url: string;
  source_title?: string;
  source_date?: string | null;
};
export type Stage1Output = {
  candidates: Stage1Candidate[];
  citations: string[];  // from r.citations
  raw: any;
};
```

## Stage 2 — Qualification (Claude)

**Model:** `settings.triageModel || 'claude-sonnet-4-6'`. Sonnet is the
right default — it's the structured-reasoning workhorse. Opus is overkill
and 5× the cost. Haiku is too small for this judgment task.

Reuse `complete()` from `llm.ts` for the Anthropic call (or `completeJson`
for direct JSON return). Pass `stage: 'deep_scan_qualify'` for telemetry.

**Stage tag:** `'deep_scan_qualify'` (new).

**Token budget:** `maxTokens: 6000`. Plenty for structured filter output.

**Prompt structure:**

```
SYSTEM (~120 words):
You are a senior B2B sales qualifier. You receive a list of raw lead
candidates surfaced by a research pass, and you decide which deserve to
land in the sales pipeline.

Be skeptical but not lazy. The research pass cast a wide net by design.
Your job is to filter, not to refuse. If a candidate clearly fits the
brand's ICP and the event is a credible buying signal, keep it. If it
doesn't fit, drop it.

You do NOT do web search. You work with the candidate list you're given.
If a candidate's company / event description is too thin to judge, drop
it as unqualified rather than guessing.

Return JSON matching the schema. Empty array is valid only when no
candidate qualifies after honest evaluation.

USER (per product):
# Brand
[brand dossier — same as Stage 1]

# Product
[product dossier — same as Stage 1]

# Target ICP (apply this strictly)
[brand.target_icp]

# Hard constraints from user
[buildGuardrails(productId) — global + product-scoped rules]

# Brand-self hygiene
[buildOwnBrandsBlock(brands) — never qualify our own brands as customers]

# Recently disqualified by user (apply same judgment)
[buildDisqualificationsBlock(product.id, 8)]

# Recently surfaced in pipeline (don't duplicate)
For dedupe — these companies have produced opportunities in the last 30
days for this product. Skip new candidates whose company name matches:
[list of distinct company names from opportunities WHERE product_id = ?
 AND created_at > date('now','-30 days')]

# Raw candidates from Stage 1
[Stage 1's candidates array, formatted as a numbered list]

# Task
For each candidate decide:
1. Does the company plausibly fit our target ICP? (size, geo, sector,
   maturity, buying-team persona)
2. Is the described event a credible buying signal for THIS specific
   product, not just general news?
3. Does the candidate pass the hard constraints (include/exclude rules)?
4. Is the company NOT one of our own brands or a near-match thereof?
5. Is the company NOT already in our pipeline for this product in the
   last 30 days?
6. Confidence 0..1 — be honest. Most should be 0.4–0.7 unless the fit
   is obvious.

For survivors, produce the full opportunity record. For each rejected
candidate, briefly note why (helpful for telemetry).

Return JSON:
{
  "opportunities": [ ...full schema records... ],
  "rejected": [ { "company": "...", "reason": "..." } ]
}
```

**Output schema** (full OPPS_SCHEMA from scanner.ts; reuse the existing
constant — DON'T duplicate). Plus `rejected` array for telemetry.

**Return type:**
```ts
export type Stage2Output = {
  opportunities: PplxOpportunity[];  // existing type
  rejected: { company: string; reason: string }[];
};
```

## Orchestration

In `scanner.ts`, the new `runDeepScan` orchestrates the two stages per product:

```ts
for (const product of scanProducts) {
  const brand = brands.find(b => b.id === product.brand_id)!;
  log(`Product scan (two-stage): ${brand.name} / ${product.name}`);

  // Stage 1
  const stage1 = await stage1Discovery(brand, product, settings, log);
  log(`  Stage 1: ${stage1.candidates.length} raw candidates, ${stage1.citations.length} citations`);
  if (stage1.candidates.length === 0) {
    log('  → 0 raw candidates from discovery — nothing to qualify');
    continue;
  }

  // Stage 2
  const recentInPipeline = db.prepare(`
    SELECT DISTINCT company FROM opportunities
    WHERE product_id = ? AND datetime(created_at) > datetime('now', '-30 days')
  `).all(product.id);
  const stage2 = await stage2Qualify(stage1, brand, product, recentInPipeline, settings, log);
  log(`  Stage 2: ${stage2.opportunities.length} qualified, ${stage2.rejected.length} rejected`);

  // Insert (reuses the existing v1.8.4 insertCandidates with all its defenses)
  const inserted = insertCandidates(
    stage2.opportunities,
    { brand, product, sourceLabel: `auto:${product.name}`, brands, citations: stage1.citations },
    { settings, seenStmt, insertSeen, insertOpp, log }
  );
  created += inserted;
  scanned += stage1.candidates.length;

  // Cross-match still runs after Pass 1 inserts (v1.7.0 behavior preserved)
  if (inserted > 0 && settings.crossMatchEnabled) {
    const xmCount = await crossMatchRecent(product.id, scanProducts, brands, log);
    created += xmCount;
  }
}
```

`runDeepScan()` checks the new setting:

```ts
export async function runDeepScan() {
  const settings = getSettings();
  if (settings.deepScanTwoStage) {
    return runDeepScanTwoStage();  // new path
  }
  // Fallback to v1.8.7 monolithic path for safety
  return runScan({
    model: settings.deepScanModel || 'sonar-deep-research',
    stage: 'deep_scan',
    kind: 'deep',
    maxTokens: 24000,
    label: '== Deep Research scan (single-stage fallback) ==',
    skipCustomTopics: true
  });
}
```

## Cost analysis

Per product per deep scan (typical):
| Stage | Model | Input tokens | Output tokens | Cost |
|---|---|---|---|---|
| Stage 1 | sonar-deep-research | ~6K | ~8K | ~$0.20–0.30 |
| Stage 2 | claude-sonnet-4-6 | ~5K | ~2K | ~$0.045 |
| **Total per product** | | | | **~$0.25–0.35** |

Current v1.8.7 per product: ~$0.20–0.40. Roughly the same — possibly slightly
cheaper because Claude Sonnet is cheap and the Perplexity call's structured-
output overhead drops.

For a 3-product portfolio twice daily: ~$1.50–2.10/day. Reasonable.

## Risks + mitigations

1. **Claude Stage 2 might also be too strict.** If qualification rate is
   below 10% (e.g., 30 raw → 1 qualified), Sonnet's filter is too tight.
   Lower the SYSTEM prompt's skepticism, or switch default to Opus.
2. **Stage 1 might still return generic guides.** If so, the prompt
   refactor needs further work — possibly explicit search-query hints per
   signal. Diagnose via Stage 1's raw candidate list.
3. **The `deepScanTwoStage: false` fallback** lets the user revert
   instantly to v1.8.7's behaviour if v1.9.0 underperforms. Don't remove
   the fallback path until v1.10 at earliest.

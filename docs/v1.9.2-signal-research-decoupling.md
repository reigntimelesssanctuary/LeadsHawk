# v1.9.2 — Signal research as a separate job

**Status:** Spec'd, not yet implemented. Drafted 2026-05-27. Awaiting user
sign-off. Part of a planned 3-release sequence:

- **v1.9.2 (this spec)** — pull signal research out of brand/product dossier
  research; introduce dossier-feedback infrastructure
- **v1.10.0** — Opus Stage 2 (verify) + Stage 3 (strategic) for brand AND
  product dossier research; feedback-aware re-research UX
- **v1.10.1** — Stage 4 fetch-and-verify

## Why this exists

Signal research is currently buried inside brand and product dossier research.
That couples two operations that shouldn't be coupled:

- Dossier research is *foundational, infrequent, expensive*. Run once, re-run
  when knowledge base changes substantially. We're about to make it more
  expensive in v1.10.0 with the Opus verification layer.
- Signal research is *tactical, iterative, cheap*. Signals change with market
  cycles. You want to refresh them often without paying the cost or the
  latency of a full dossier re-research.

Two concrete problems with the current coupling:

1. **You can't iterate on signals without re-paying for dossier research.**
   Today, clicking "Run research" on a product regenerates the entire dossier
   even if the only thing you wanted to refresh was the signal list. The
   `refreshProductSignals` shortcut exists but is buried on the product card
   and isn't symmetric for brand-level signals.
2. **There's no place for signal research to accept user feedback.** Brand
   owners reviewing the dossier may say "the signals are too generic" or
   "you missed lease-renewals as a buying trigger" — there's no clean UX
   for incorporating that feedback into a targeted re-run.

User's decision (2026-05-27): pull signal research into its own job, surfaced
from the Signal Config tab, with per-brand and per-product buttons. Symmetric
with how dossier research is surfaced from the Brands & Products tab.

## Architecture overview

```
                       ┌─────────────────────────────────────────┐
                       │  Brands & Products tab                  │
                       │  • Run Brand Research / Re-research     │
                       │  • Run product research / Re-research   │
                       │  (Dossier ONLY — no signals here)        │
                       └─────────────────────────────────────────┘

                       ┌─────────────────────────────────────────┐
                       │  Signal Config tab                       │
                       │  ── Brand signals section ──             │
                       │   • For each brand: Run signal research  │
                       │   • Re-research with feedback            │
                       │  ── Product signals section ──           │
                       │   • For each product: Run signal research│
                       │   • Re-research with feedback            │
                       └─────────────────────────────────────────┘

                       ┌─────────────────────────────────────────┐
                       │  Downstream consumers (unchanged)        │
                       │  • Scanner Pass 1 reads products.signals │
                       │  • Scanner Pass 1 reads brands.signals   │
                       │  • Live monitor pre-filter embeds        │
                       │    products.signals                      │
                       │  • Cross-match (v1.7) uses cached embeds │
                       │  • v1.9 Stage 1 discovery uses           │
                       │    products.signals as orientation       │
                       └─────────────────────────────────────────┘
```

The columns `brands.signals` and `products.signals` stay where they are.
Downstream consumers don't change. What changes is **who writes those
columns** — no longer a side-effect of dossier research, but a dedicated
Signal Config job.

## What changes

### 1. Schema changes (Perplexity prompts)

In `src/main/research.ts`:

- `RESEARCH_SCHEMA` (product dossier): **remove** the `signals` required field.
  Update the SYSTEM prompt accordingly so the model stops generating signals.
- `BRAND_RESEARCH_SCHEMA` (brand dossier): **remove** the `signals` required
  field. Update the SYSTEM prompt accordingly.
- `researchProduct()`: don't write `signals` in the UPDATE statement; leave
  the column untouched (preserves whatever was there before).
- `researchBrand()`: same — don't write `signals` in the UPDATE statement.

**Behavioural implication:** existing brands/products keep their currently
stored signals after v1.9.2 is installed. Future re-researches won't touch
them. New brands/products start with `NULL` signals until the user runs
Signal Config → Research signals on them.

### 2. New signal-research functions

In a new module `src/main/signal-research.ts`:

```ts
export async function researchProductSignals(
  productId: number,
  options?: { feedback?: string }
): Promise<Product>
```

Lift the existing `refreshProductSignals()` body (currently in `research.ts`),
move it here, and extend with:
- Optional `feedback` parameter
- When `feedback` is provided, inject a "Reviewer feedback to incorporate"
  block above the task instructions
- Also inject the full feedback history for this product (latest 20 entries,
  newest first) so prior corrections aren't lost across iterations
- Telemetry tag changes from `'refresh_signals'` → `'product_signals'`

```ts
export async function researchBrandSignals(
  brandId: number,
  options?: { feedback?: string }
): Promise<Brand>
```

New function — mirror of the product version, scoped to brand-level signals.
Uses `sonar-pro` by default (cheap), with brand dossier as context (similar
shape to `researchProductSignals`). Schema is `{ signals: string }` only.
Telemetry tag: `'brand_signals'`.

Both functions also call `embedSignalsForProduct(id)` after writing product
signals so the live-monitor pre-filter stays in sync. Brand signals don't
need embedding — the live monitor matches on product signals only today.

### 3. Feedback infrastructure (introduced in v1.9.2, reused by v1.10.0)

New table in `src/main/db.ts`:

```sql
CREATE TABLE IF NOT EXISTS dossier_feedback (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  target_kind TEXT NOT NULL,
  -- 'brand' | 'product' | 'brand_signals' | 'product_signals'
  target_id INTEGER NOT NULL,
  feedback TEXT NOT NULL,
  applied_at TEXT,
  -- when the research run that used this feedback completed (NULL = not yet applied or pending)
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_feedback_target
  ON dossier_feedback(target_kind, target_id, created_at DESC);
```

New module `src/main/feedback.ts`:

```ts
export type FeedbackTargetKind =
  | 'brand'           // v1.10.0 will use this
  | 'product'         // v1.10.0 will use this
  | 'brand_signals'   // v1.9.2
  | 'product_signals';// v1.9.2

export type DossierFeedback = {
  id: number;
  target_kind: FeedbackTargetKind;
  target_id: number;
  feedback: string;
  applied_at: string | null;
  created_at: string;
};

export function listFeedback(
  kind: FeedbackTargetKind,
  targetId: number,
  limit?: number
): DossierFeedback[];

export function addFeedback(
  kind: FeedbackTargetKind,
  targetId: number,
  feedback: string
): number;  // returns inserted id

export function markFeedbackApplied(id: number): void;

/**
 * Render feedback history as a prompt-ready block.
 * Honours the 4000-char per-entry cap (already enforced on insert)
 * plus a total-block cap of 16000 chars.
 *
 * Returns empty string if no feedback exists.
 */
export function buildFeedbackBlock(
  kind: FeedbackTargetKind,
  targetId: number
): string;
```

Feedback length enforcement:
- **Per-submission cap: 4000 chars** (validated server-side in
  `addFeedback`; throw if exceeded with a clear error so the renderer can
  surface it)
- **Total feedback-block cap: 16000 chars** (newest-first, truncate older
  entries to fit; the truncation happens in `buildFeedbackBlock`)

### 6. Signal Config UI

`src/renderer/src/pages/SignalConfig.tsx` gets two new top sections, inserted
above the existing "Auto-derived signals" section:

#### Brand signals section

A list of every researched brand (status `'ready'`). Each row shows:
- Brand name + chip with current state (`no signals yet` / `N signals` / `researching…`)
- Inline preview of current signals (collapsible, first 3 shown)
- Two buttons:
  - **Research signals** — runs `window.lh.brands.researchSignals(id)`. Disabled
    while a research run is in flight for any brand/product (use a global
    flag in the SignalConfig component state).
  - **Re-research with feedback** — opens a modal.

#### Product signals section

Identical structure to brand signals but per-product, rolling up by brand
(so users see Brand A → its products, Brand B → its products). The existing
"Auto-derived signals (primary)" section in SignalConfig.tsx is kept but
its "Research signals" buttons move here so the page has one canonical
signal-management area.

#### Re-research-with-feedback modal

A shared modal component `SignalFeedbackModal` opened by both sections.

Layout:
- Title: `Re-research signals for [Brand|Product Name]`
- Feedback history (read-only, scrollable):
  - Each entry: date + truncated preview (collapsed by default; click to expand)
  - "No prior feedback" empty state
- New feedback textarea:
  - Placeholder: `Tell the model what to change. E.g. "Focus on lease renewals — exec changes are noise for this product."`
  - Char counter `N / 4000` — turns red at 3800+
  - Disabled submit if empty or over 4000
- Help text: `Your feedback will be incorporated into the next signal-research run for [Brand|Product]. Prior feedback is automatically re-applied so corrections don't get lost across iterations.`
- Footer buttons: `Cancel` | `Submit and re-research`

On submit:
1. Call `feedback.add(kind, targetId, text)` to persist
2. Call `brands.researchSignals(id, { feedback: text })` or `products.researchSignals(id, { feedback: text })`
3. On success, call `feedback.markApplied(id)` for the inserted feedback row
4. Refresh signal list

## Cost analysis

No change. Signal research today uses Perplexity `sonar-pro` with
`max_tokens: 1500` — typically ~$0.005–0.02 per call. Adding the
brand-symmetric call and the feedback infrastructure doesn't change
per-call cost.

For a 10-brand + 30-product portfolio refreshed monthly: ~$0.40/month
total signal-research spend. Negligible.

## Risks + mitigations

1. **Existing data — what happens to brands/products with no signals after
   v1.9.2 installs?** Answer: they keep whatever was previously written to
   the column. Nothing is wiped. Fresh research won't overwrite. The
   "signals not researched yet" banner only fires on rows with NULL or
   empty signals.
2. **Feedback length** — capped at 4000 chars per submission, total block
   capped at 16K chars. If the user truly needs to convey a 10K-char
   correction they can split it across two submissions (history preserves
   both).
3. **`refresh_signals` LlmStage rows in existing spend** — kept in the enum
   so the spend dashboard still labels them correctly. New runs use the
   new tags.
4. **Concurrency** — if a user clicks "Research signals" on multiple
   brands/products at once, each call is independent (they go through
   separate IPC handlers, separate Perplexity jobs). No locking needed.
   The UI disables individual buttons while their own call is in flight;
   a global "any research in flight?" disable is overkill.

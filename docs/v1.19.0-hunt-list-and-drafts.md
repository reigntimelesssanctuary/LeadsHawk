# v1.19.0 — Contact search + signal-grounded drafts (Phase 1 of outbound)

**Status:** Spec'd, not yet implemented. Drafted 2026-06-08, rewritten
2026-06-08 incorporating operator feedback. Execute as the next release
after v1.18.0.

**Phase context:** Phase 1 of a three-phase outbound transformation.
Phase 2 (v1.20+) brings **Smartlead** in for automated sending, reply
detection, and sequenced cascade through ranked contacts. Phase 3 may
bring CRM push, or may never happen — LeadsHawk stays the operator's
internal cockpit. Clients never log in; they receive booked meetings
via email.

---

## 1. Why this exists

LeadsHawk through v1.18.0 stops at "we surfaced this opportunity; here's
the brief." Acting on that requires the operator to hunt contacts,
decide who to email first, and write a personalised first-touch — all
mechanical given the dossier + qualifying event context LeadsHawk
already holds, but all happening outside the app today.

v1.19.0 brings those three steps inside, **manually triggered per
opportunity by the operator**, no automatic firing:

1. Click **Search contacts** on an opportunity (single or multi-select).
2. A reasoning model decides what *archetype* of contact would care
   about THIS event for THIS product, then queries Apollo with those
   filters → returns 3–5 contacts ranked by fit.
3. Operator clicks **Draft email** on the top-ranked contact; an
   extended-thinking model writes the first-touch grounded in the
   event + the contact's role.
4. Operator reviews, edits, copies, sends from their own email tool.

Phase 2 takes over from step 4 — pushing approved drafts to Smartlead,
which handles delivery + sequenced cascade through the ranked contacts.
v1.19 leaves the schema and IPC seams for that.

## 2. Architecture overview

```
Dashboard row action: [Search contacts]  ─────────────────┐
                                                          │
  (or bulk action on multi-select)                        │
                                                          ▼
                                    searchContactsForOpportunity(oppId)
                                          │
                                          ├── stage1 — Archetype reasoning
                                          │     ↓ Claude Sonnet 4.6 (no extended thinking)
                                          │     ↓ Returns: {
                                          │     ↓   target_seniorities[], target_titles[],
                                          │     ↓   target_departments[], anti_patterns[],
                                          │     ↓   reasoning
                                          │     ↓ }
                                          │     ↓ Persisted: contact_searches row
                                          │
                                          ├── stage2 — Apollo search
                                          │     ↓ POST /v1/mixed_people/search
                                          │     ↓ Filters: org_name + seniorities + titles
                                          │     ↓ per_page=25 (Apollo returns up to 25
                                          │     ↓   passable matches; we rank + take top 3-5)
                                          │
                                          └── stage3 — Rank + persist
                                                ↓ Pure scoring (no LLM)
                                                ↓ Top 3-5 inserted as contacts rows
                                                ↓ All rows start in contact_status='pending'


Opportunity Detail → Hunt list section, per-contact [Draft email] button
                                                          │
                                                          ▼
                                              draftEmailForContact(contactId)
                                                ↓ Claude Opus 4.7 + extended thinking
                                                ↓ Returns: { subject, body, reasoning_trace,
                                                ↓            one_line_why }
                                                ↓ Persisted as contact_drafts row
                                                ↓   (draft_version auto-incremented if a
                                                ↓    draft already exists for this contact)
                                                ↓ Contact status pending → drafted
```

Two LLM stages, two storage tables. Manual everywhere. No auto-firing.

## 4. Apollo integration

### 4.1 Why Apollo

Decided 2026-06-08. Chosen for:

- **Self-serve onboarding** — operator (non-coder) signs up in 5 min,
  generates an API key in their dashboard, pastes into LeadsHawk
  Settings. No domain verification, no CRM sync, no sales call.
- **Single-call breadth** — `/mixed_people/search` returns name, title,
  seniority, work email + status, LinkedIn URL, sometimes personal
  email. Nothing else at this price tier has that depth.
- **Sane pricing** — Free tier = 60 credits/mo (covers initial testing);
  Starter = $49/mo for 1,000 credits ≈ 200 opportunities at 5
  contacts/opp.
- **Stable REST API** — well-documented, predictable error shapes.

Rejected alternatives: Hunter (email-only, no role data), RocketReach
(API behind pricier plans), Clearbit (now requires HubSpot account),
ZoomInfo (enterprise sales motion).

### 4.2 Endpoints used

| Endpoint | Purpose | Cost |
|---|---|---|
| `POST /v1/mixed_people/search` | Find contacts at a company filtered by seniority + title patterns | 1 credit per result returned |
| `GET /v1/auth/health` | Validate API key on Settings save | Free |

We rely on `mixed_people/search` exclusively for v1.19 — it supports
filtering by `organization_names[]`, `person_seniorities[]`,
`person_titles[]`, and `q_keywords` in a single call. Per-opportunity
budget: target 3–5 results = 3–5 credits.

Email is returned inline when available. Free tier may return email
status `'unavailable'` for some contacts — we mark these accordingly
and rank them lower but still surface them (operator can manually
find the email if needed).

### 4.3 Rate limits + retry

Apollo's REST limit on Starter is ~600 requests/minute. Our cadence
(worst case 50 opps/day × 1 search = 50 requests) sits orders of
magnitude below. One retry on 429 / network error; no exponential
backoff needed initially.

### 4.4 API key + secrets

Stored in `electron-store` settings.json under `apolloApiKey`. Validated
on save via the `auth/health` endpoint. No env vars, no separate config
file. The Settings page gets a new "Contact API (Apollo)" card.

## 5. Contact archetype reasoning (Stage 1 of search)

**Why this exists:** different events at different companies call for
different contact archetypes. Generic title filters (always "CIO",
always "VP IT") miss the specificity that earns replies.

Examples of the right archetype shifting by event:
- "Acme opens Singapore HQ" + networking gear → Facilities Lead +
  IT Infrastructure Director (NOT generic CIO)
- "Acme suffers ransomware breach" + security product → CISO + SOC
  Head (NOT CIO)
- "Acme announces Series C" + analytics product → CFO + VP Data
  (NOT IT)

A pure static title list can't do this. A small reasoning model can.

### 5.1 Model + cost

**Claude Sonnet 4.6**, no extended thinking. The task is bounded
(pick from a known taxonomy), output is structured (~200 tokens),
latency is sub-3s. Cost: ~$0.01 per search.

Reserved for Sonnet not Opus because the reasoning here is well-bounded
and structured. Opus is conserved for the drafting stage where writing
quality dominates.

### 5.2 System prompt (sketch)

```
You are a B2B account researcher. Given a brand's positioning, one of
its products, and a specific recent event at a target company, your
job is to decide WHAT KIND OF CONTACT at that company would most
likely care about THIS event for THIS product.

Think about: who in the buying committee owns the outcome the event
implies? Who has the budget? Who feels the pain the event creates?

Output a structured query plan that a contact-data API can use to find
those people. Be specific — generic "CIO" answers waste outreach
credits. Match seniority to deal size and product complexity.

Anti-patterns are roles that look superficially relevant but are wrong
for this context (e.g. "Marketing" for an IT infrastructure event).

Return strictly valid JSON only.
```

### 5.3 Output schema

```json
{
  "target_seniorities": ["c_suite", "vp", "director"],
  "target_titles": [
    "Head of Infrastructure",
    "VP Engineering",
    "Director of Network Operations"
  ],
  "target_departments": ["engineering", "it_operations"],
  "anti_patterns": ["sales", "marketing", "hr", "legal"],
  "reasoning": "Singapore HQ expansion creates networking + facilities pain; ICP suggests director-band buyers own this category at mid-market firms."
}
```

`target_seniorities` enum values mirror Apollo's: `c_suite`, `vp`,
`director`, `head`, `manager`, `senior`, `entry`, `owner`, `partner`,
`founder`.

### 5.4 Failure handling

- Sonnet error → search fails, opp's `hunt_status='search_failed'`,
  operator sees a chip + can retry.
- Unparseable JSON → one retry; if still unparseable, log and fail
  cleanly.
- Empty `target_titles` → fall back to a generic default (top three
  seniorities × broad role pattern derived from `brand.target_icp`)
  so we still produce *some* archetype rather than skipping the
  Apollo call.

### 5.5 Persisted as audit record

Every archetype invocation writes a row to `contact_searches`
(append-only — keeps the historical archetype across re-searches so
the operator can see how the model's thinking evolved).

## 6. Apollo search (Stage 2)

Built directly from the Stage 1 archetype:

```
POST /v1/mixed_people/search
{
  "organization_names": ["Acme Corp"],
  "person_seniorities": ["c_suite", "vp", "director"],
  "person_titles": ["Head of Infrastructure", "VP Engineering", …],
  "per_page": 25
}
```

We request up to 25 candidates so the ranker has signal to choose
from. Anti-pattern roles are filtered post-search (Apollo doesn't
support exclusion filters cleanly).

Charged credits = `min(returned, 25)`. Typical real-world response
for a mid-market target is 8–15 matches; for SMBs may be 1–3.

If the result is **0 matches**, search marked
`run_status='no_contacts'`. Opportunity gets `hunt_status='no_contacts'`
chip on the Dashboard. Operator can retry later (the company may not
be in Apollo's index yet, or filters were too narrow).

## 7. Hunt ranking (Stage 3, pure logic)

Deterministic scoring against the Stage 1 archetype. No LLM call.

```
hunt_score(contact, archetype, opp) =
    0.40 × archetype_title_match    (contact.title  vs archetype.target_titles)
  + 0.25 × seniority_match          (contact.seniority vs archetype.target_seniorities)
  + 0.15 × department_match         (contact.department vs archetype.target_departments)
  - 0.10 × anti_pattern_penalty     (contact.title  vs archetype.anti_patterns)
  + 0.05 × verified_email_bonus     (contact.email_status === 'verified')
  + 0.05 × signal_keyword_match     (contact.title  vs opp.matched_signal terms)
```

All bounded 0..1; sorted descending. Take top **N** where N = `min(5,
matches)` and minimum N = 3 to declare success. Below 3 → mark
`no_contacts` (insufficient quality even if Apollo returned more).

### 7.1 Component definitions

- **archetype_title_match** — fuzzy token-overlap between contact's
  title and any of `archetype.target_titles`. Implemented as Jaccard
  similarity over lowercased word tokens.
- **seniority_match** — exact-match boolean → 1.0 if contact's
  seniority appears in `target_seniorities`, else 0.
- **department_match** — overlap between Apollo's inferred department
  string and `target_departments`. Apollo may not always populate this
  cleanly — when missing, contributes 0 (not penalised).
- **anti_pattern_penalty** — fuzzy match against any anti-pattern term;
  if matched, full 0.10 deducted.
- **verified_email_bonus** — flat 0.05 if `email_status === 'verified'`.
- **signal_keyword_match** — Substring presence of any 2+ keywords
  extracted from `opp.matched_signal + opp.headline` against the title.

### 7.2 Tunability

All weights live in `src/shared/hunt.ts` as exported constants. No
user-facing knobs. Tuning is a code change with smoke-test coverage
— same pattern as the v1.17 magnitude cap. Recommend revisiting
weights after ~50 hunts of operator-observed quality data.

### 7.3 Persistence

`rank_components` JSON stored per contact so the operator can inspect
WHY a contact landed at rank #1 vs #3.

## 8. Email draft generation

### 8.1 Manual trigger, lazy generation

**No drafts are generated at search time.** The Hunt list panel
renders 3-5 contacts with no drafts; each has its own [Draft email]
button.

Operator clicks → one Opus call → draft persisted → contact's
`contact_status` flips `pending` → `drafted`.

Subsequent click on the SAME contact's [Draft email] button opens a
confirm: *"A draft already exists. Generate an alternative version?
(The existing draft will be preserved as version 1.)"* On confirm,
the new draft is stored with `draft_version=2,3,…` and `is_active=1`
(making it the default-selected version for any future send). The
operator can switch back via a version dropdown on the contact card.

This means **no LLM spend on contacts the operator doesn't intend
to reach** — important because most opps will be acted on for the
top 1–2 contacts only.

### 8.2 Model + cost

**Claude Opus 4.7 with extended thinking enabled.**

Reasoning:
- Cold email is a one-shot high-stakes writing-and-judgment task.
- Extended thinking lets Opus reason through *what's the right angle
  for THIS person about THIS event for THIS product* before writing.
- The reasoning trace gets surfaced in the UI as **"Why this angle"** —
  the operator inspects it, decides whether to trust the draft, and
  can regenerate-with-feedback if the angle's off.

Cost: ~$0.05–0.08 per draft with extended thinking (vs ~$0.02–0.03
without). Bounded per opp by lazy generation — operator typically
drafts 1–2 contacts per opp = $0.05–0.16 per acted-on opp.

### 8.3 System prompt — cold email best practices encoded

The system prompt is the heart of draft quality. Drafted from
marketing best practices:

```
You are writing a cold first-touch email on behalf of a B2B sales
operator. The email's relevance comes entirely from grounding in a
SPECIFIC recent event at the recipient's company, framed against
the recipient's role.

Hard rules:
- Subject ≤ 60 characters. Specific. No clickbait. No emojis.
  No "RE:" or "FW:" tricks.
- Body ≤ 120 words total. 3 short paragraphs maximum.
- Open with the event (their context), NOT with yourself.
- Reference the recipient's role naturally — don't lecture them on
  what their job is.
- Show competence in 1-2 sentences (a sharp observation about the
  event's implications). Do NOT pitch the product by name in this
  first email. The goal is to earn a reply, not to sell.
- Single specific CTA: "Open to a brief call next week to compare
  notes?" — lower commitment than "demo" or "meeting".
- End with a single clear question.

Forbidden phrases (will get the email instantly discarded by
sophisticated recipients):
- "I hope this email finds you well"
- "I wanted to reach out"
- "Circling back"
- "Just following up"
- "Synergies", "alignment", "leverage" used loosely
- Mail-merge artefacts like "{First Name}" or "[Company]"

Tone: confident, specific, brief. Like one operator emailing
another — not like marketing.

Format: mobile-readable. Most B2B recipients open on phone first.
Short paragraphs. No HTML formatting. No attachments. No images.

DO NOT include a signature — the sender's email platform appends
one automatically. End the body at the question, nothing after.

Use extended thinking to:
1. Identify the SPECIFIC angle that ties this event to this product
   for this role.
2. Reject the most generic version of that angle.
3. Find one sharp observation worth saying.
4. Draft the email.

Return strictly valid JSON only:
{
  "subject": "...",
  "body": "...",
  "one_line_why": "<short summary of the angle choice>"
}

The extended-thinking trace is returned separately by the model and
will be persisted for operator inspection.
```

### 8.4 Regenerate with feedback

Reuses the v1.9.2 feedback infrastructure (`dossier_feedback` table)
pattern, with a new `target_kind = 'contact_draft'`. Operator clicks
"Regenerate", optionally adds one-line feedback (e.g. "less corporate,
more direct"), the next draft incorporates it. Old draft stays
preserved as a prior version.

### 8.5 Inline edit

Operator can edit subject + body inline. Saves to DB, marks
`human_edited=1`. Edited drafts always win over their LLM source
when present (Phase 2's send step will prefer them).

### 8.6 Signature — managed by Smartlead, not LeadsHawk

Smartlead lets you configure a signature per sender mailbox. LeadsHawk
drafts the body only, no signature appended. For Phase 1 (copy-paste
to operator's own email tool), the operator's email client appends
their own signature on send. For Phase 2, Smartlead handles it.

**No "operator signature" field in LeadsHawk Settings.** Clean
separation of concerns.

## 9. Schema additions

All idempotent migrations via `addColumnIfMissing` /
`CREATE TABLE IF NOT EXISTS`. No backfill — `contact_searches`,
`contacts`, and `contact_drafts` are new tables; old opportunities
simply have no rows until searched.

```sql
-- v1.19.0
-- Audit log of every archetype-reasoning invocation. Append-only;
-- re-searches add new rows so the operator can see how the archetype
-- shifted across attempts.
CREATE TABLE IF NOT EXISTS contact_searches (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  opportunity_id  INTEGER NOT NULL REFERENCES opportunities(id) ON DELETE CASCADE,
  archetype_json  TEXT NOT NULL,         -- Sonnet's structured output (see §5.3)
  reasoning       TEXT,                  -- Sonnet's one-line rationale
  contacts_found  INTEGER NOT NULL DEFAULT 0,
  apollo_credits  INTEGER NOT NULL DEFAULT 0,
  llm_cost        REAL NOT NULL DEFAULT 0,
  run_at          TEXT NOT NULL DEFAULT (datetime('now')),
  run_status      TEXT NOT NULL DEFAULT 'pending'
    -- pending | completed | no_contacts | search_failed
);
CREATE INDEX IF NOT EXISTS idx_contact_searches_opp ON contact_searches(opportunity_id);

CREATE TABLE IF NOT EXISTS contacts (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  opportunity_id  INTEGER NOT NULL REFERENCES opportunities(id) ON DELETE CASCADE,
  search_id       INTEGER REFERENCES contact_searches(id) ON DELETE SET NULL,
  -- Apollo fields
  apollo_id       TEXT,
  full_name       TEXT NOT NULL,
  first_name      TEXT,
  last_name       TEXT,
  title           TEXT,
  seniority       TEXT,
  department      TEXT,
  email           TEXT,
  email_status    TEXT,                  -- verified | guessed | unavailable | unverified
  linkedin_url    TEXT,
  -- Ranking
  hunt_rank       INTEGER NOT NULL,
  hunt_score      REAL NOT NULL,
  rank_components TEXT,                  -- JSON breakdown
  -- State machine
  contact_status  TEXT NOT NULL DEFAULT 'pending',
  -- v1.19 states:
  --   pending  (just discovered, no draft yet)
  --   drafted  (≥1 draft generated, not sent)
  --   sent     (operator manually marked sent — Phase 1)
  --   skipped  (operator explicitly skipped)
  --   failed   (draft generation errored out)
  -- v1.20+ states (Phase 2 — additive, no schema change):
  --   sequencing_active            (pushed to Smartlead, in flight)
  --   sequence_paused_awaiting_decision  (any reply received, operator decides stop/continue)
  --   replied                      (operator confirmed real reply)
  --   bounced_hard                 (hard bounce, sequence escalated)
  --   bounced_soft                 (soft bounce, awaiting retry)
  --   unsubscribed                 (recipient opted out)
  --   sequence_complete_no_response (wait window exhausted)
  -- Tracking
  marked_sent_at  TEXT,                  -- when operator clicked "Mark sent" (Phase 1)
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_contacts_opp ON contacts(opportunity_id);
CREATE INDEX IF NOT EXISTS idx_contacts_rank ON contacts(opportunity_id, hunt_rank);
CREATE INDEX IF NOT EXISTS idx_contacts_status ON contacts(contact_status);
-- Per-opp Apollo dedup so re-search doesn't double-insert the same person
CREATE UNIQUE INDEX IF NOT EXISTS uniq_contacts_apollo_per_opp
  ON contacts(opportunity_id, apollo_id) WHERE apollo_id IS NOT NULL;

-- Per-contact draft history. Multiple versions supported; is_active
-- marks the currently-selected draft for any send action.
CREATE TABLE IF NOT EXISTS contact_drafts (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  contact_id      INTEGER NOT NULL REFERENCES contacts(id) ON DELETE CASCADE,
  draft_version   INTEGER NOT NULL,      -- 1, 2, 3, ... per contact
  subject         TEXT NOT NULL,
  body            TEXT NOT NULL,
  reasoning_trace TEXT,                  -- Opus extended-thinking trace (long)
  one_line_why    TEXT,                  -- short summary for the UI chip
  human_edited    INTEGER NOT NULL DEFAULT 0,
  is_active       INTEGER NOT NULL DEFAULT 1,
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_drafts_contact ON contact_drafts(contact_id);
-- Exactly one is_active=1 row per contact (enforces "current draft")
CREATE UNIQUE INDEX IF NOT EXISTS uniq_active_draft_per_contact
  ON contact_drafts(contact_id) WHERE is_active = 1;

-- Per-opportunity hunt status for the Dashboard chip
ALTER TABLE opportunities ADD COLUMN hunt_status TEXT;
-- v1.19 values:
--   NULL              (never searched)
--   searching         (Apollo call in flight)
--   hunted            (≥1 contact returned + ranked)
--   no_contacts       (Apollo returned <3 matches)
--   search_failed     (Sonnet or Apollo errored)
-- v1.20+ extends with: sequencing | meeting_booked | exhausted | handed_off
```

The `ALTER TABLE` is done via the existing `addColumnIfMissing` helper
to keep the migration idempotent on upgrade.

## 10. New IPC endpoints

All preloaded under `window.lh.contacts.*` and
`window.lh.settings.validateApolloKey`.

| IPC | Purpose |
|---|---|
| `contacts:search(oppId)` | Run the Sonnet archetype → Apollo search → rank → persist pipeline for ONE opp. Returns the persisted `Contact[]` plus the `ContactSearch` audit row. Idempotent at the Apollo level (dedup by `apollo_id`); subsequent calls smart-replace pending contacts (see §11). |
| `contacts:searchBatch(oppIds[])` | Loops `contacts:search` over the array sequentially. Returns per-opp success/fail outcomes. |
| `contacts:listForOpp(oppId)` | Returns ranked contacts + their active drafts for the Hunt list panel. |
| `contacts:draftEmail(contactId, opts?)` | Run Opus extended-thinking draft for ONE contact. `opts.feedback` enables regenerate-with-feedback (creates a new `draft_version`). |
| `contacts:setActiveDraft(contactId, draftId)` | Switch which draft is `is_active=1` for that contact. |
| `contacts:updateDraft(draftId, subject, body)` | Inline human edit. Sets `human_edited=1`. |
| `contacts:markSent(contactId)` | Phase 1 manual gate: `contact_status='sent'`, `marked_sent_at=now`. |
| `contacts:skip(contactId)` | `contact_status='skipped'`. |
| `contacts:unskip(contactId)` | Reverses a skip back to its prior state (pending or drafted depending on whether drafts exist). |
| `contacts:delete(contactId)` | Hard-delete (Apollo returned an obviously wrong contact). Cascades drafts. |
| `settings:validateApolloKey(key)` | Round-trips `auth/health`; returns `{ ok, error?, creditsRemaining? }`. |

## 11. Re-search behaviour (smart-replace)

When the operator clicks **Re-search** on an opp that has previously
been searched:

1. Apollo is called fresh with a re-derived archetype (Sonnet runs
   again — operator may have updated dossier/scan-rules in between,
   so the archetype can legitimately shift).
2. New contacts are matched against existing contacts by `apollo_id`:
   - Already-present apollo_ids: keep existing row (preserves any
     drafts, sent status, etc.)
   - New apollo_ids: insert as new contact rows; rank from
     `max(existing.hunt_rank) + 1` to keep ordering stable
3. Pending contacts NOT returned by the new search get **removed**.
   Contacts in non-pending states (`drafted`, `sent`, `skipped`,
   anything Phase-2 set) are **preserved** regardless.
4. Confirm dialog before commit:
   ```
   Re-searching will:
   - Keep N contacts in non-pending state (drafted/sent/skipped)
   - Replace M pending contacts with up to 5 fresh ones
   Continue?
   ```

Re-search always writes a new `contact_searches` audit row.

## 18. Phase 2 (v1.20+) preview — seams left in v1.19

The following are NOT built in v1.19 but the schema/IPC choices in
v1.19 are designed so v1.20 doesn't require a rewrite:

- **Smartlead push** — `contacts:approveAndSend(contactId)` becomes a
  new IPC that wraps `markSent` plus pushes the active draft +
  contact + sender mailbox into a Smartlead campaign via their API.
  Existing `markSent` remains as the manual fallback.
- **Sequencing engine** — node-cron job that polls each
  `sequencing_active` contact for the wait-interval expiry. On expiry
  with no reply, calls `draftEmailForContact(nextContactId)` then
  `approveAndSend`. Cascades through `hunt_rank` order.
- **Hard-bounce escalation** — Smartlead webhook handler. On hard
  bounce: `contact_status = 'bounced_hard'` → automatically trigger
  draft + send for the next-ranked pending contact (no wait).
- **Reply pause** — Smartlead webhook handler. On ANY reply:
  `contact_status = 'sequence_paused_awaiting_decision'`. Operator
  sees a new "Awaiting your call" chip on the Dashboard, opens the
  opp, reads the reply (synced from Smartlead), clicks
  **Stop sequence** (it was a real reply → done; lifecycle event
  `engaged`) or **Continue sequence** (it was an OOO / wrong-person
  / autoresponder → cascade to next contact).
- **Max-cadence ceiling** — sequencing engine enforces 5 contacts ×
  N-business-days = ~25-day ceiling per opportunity to protect
  sender reputation.
- **Cross-contact dedup at Apollo level** — already handled in v1.19
  via `uniq_contacts_apollo_per_opp` index.
- **`hunt_status` extension values** — `sequencing` / `meeting_booked`
  / `exhausted` / `handed_off` — additive, no schema change.

No premature implementation. Just preserve the seams.
